## Execution Model

### Single-Session Architecture ⚡

- **Login Once**: The suite logs in only once in `beforeAll()` (not per-test).
- **Shared Browser Session**: All contract test cases run in the same browser session across all three describe blocks.
- **No Logout**: The browser session persists until `test.afterAll()` completes (after all tests finish).
- **Smart Navigation**: `beforeEach()` only navigates to Deals page if not already there (checks URL).
- **No Redundant Setup**: Company and Property creation happens ONLY if PAT-prefixed resources don't already exist.

### Performance Optimizations

1. **Reuse PAT Resources**: Before creating company/property, the suite searches for existing ones with "PAT_" prefix (auto-generated by tests). If found, reuses them immediately.
2. **Skip Unnecessary Creations**: Eliminates redundant company/property creation on repeated runs (~60% faster).
3. **Single Deal Reuse**: One deal is created and reused across:
   - Smoke tests (TC-CONTRACT-001 to TC-CONTRACT-022)
   - E2E full flow (TC-CONTRACT-E2E-001 to TC-CONTRACT-E2E-018)
   - Service deletion tests (TC-CONTRACT-DELETE-001 to TC-CONTRACT-DELETE-010)

### Test Organization

- **3 describe blocks** — all share the same session
  - `Contract & Terms Module — Complete Suite` (smoke tests)
  - `Contract Module — E2E Full Create & Publish` (full proposal→publish flow)
  - `Contract Module — Service Deletion & Total Updates` (service management)
- **62 test cases** total (expanded from original 40 with new optimization tests)
- **Serial execution** — tests run sequentially for state consistency

## Dynamic Test Data

- **Company Resolution**:
  1. Search for existing company with "PAT_" prefix
  2. If found → reuse it
  3. If not found → create `PAT_{timestamp}`

- **Property Resolution**:
  1. Search for existing property with "PAT_" prefix
  2. If found (and matches company) → reuse it
  3. If not found → create `PAT_{timestamp}` for the resolved company

- **Contract Target Deal**:
  - Resolved dynamically from environment or created fresh
  - Reused across all test suites in the same session

- **Time Zone**: Eastern (for contract date calculations)
- **Stepper & Publish Data**: From `utils/contract-test-data.js`

## Assertion Requirements (CRITICAL)

**See [@.claude/skills/playwright-test-standards/SKILL.md](../../.claude/skills/playwright-test-standards/SKILL.md#6-assertion-rules) — Section 6: Assertion Rules**

⚠️ **EVERY TEST MUST HAVE MEANINGFUL ASSERTIONS** — Weak assertions (like `toBeDefined()` alone) are NOT acceptable.

Quick reference:
- ❌ `expect(total).toBeDefined()` — Only checks existence
- ✅ `expect(total).toMatch(/\$[\d,]+\.\d{2}/)` — Validates format
- ✅ `await expect(button).toBeEnabled()` — Validates state
- ✅ `await expect(toast).toHaveText(/saved/i)` — Validates user-visible outcome

When documenting test cases below, the "Expected result" field MUST describe what assertion validates the outcome.

## Manual QA Focus: Create Proposal blocked when Time Zone is not selected

Scope:
- This section covers only the Create Proposal drawer behavior for Time Zone required validation.
- Goal: verify user cannot proceed when Time Zone is missing, and required feedback is shown clearly.
- Do not use this section for stepper/device/payment flow validation.

- User is on Deal Detail page.
- `Contract & Terms` tab is open.
- A deal state is available where `Create Proposal` drawer can be opened.

### M-CONTRACT-TZ-001 | Create Proposal must be blocked when Time Zone is missing (complete validation coverage)

#### Detailed execution steps
1. Navigate to `BASE_URL` and login using valid HO credentials from `.env.uat`.
2. Open `Deals` list and open a deal where `Contract & Terms` is accessible.
3. Go to `Contract & Terms` tab and click `Create Proposal`.
4. Verify baseline drawer state:
   - Time Zone control is visible and interactive.
   - No validation error is shown before submit.
   - Current Time Zone state (pre-filled or empty) is noted.
5. Keep Time Zone empty/unselected.
6. Fill all other mandatory fields with valid data (Proposal Name, Start Date, Renewal/End Date as applicable).
7. Click `Create Proposal`.
8. Verify blocking behavior and required validation:
   - Submit is blocked.
   - Drawer remains open.
   - User does not enter stepper flow.
   - Time Zone required feedback appears near the field.
9. Verify error quality:
   - Error text clearly indicates Time Zone is required.
   - Invalid styling is visible on field/control.
10. Select a valid Time Zone option (any valid `(UTC...)` item) and submit again.
11. Verify recovery:
    - Required error clears.
    - Submit succeeds when all required inputs are valid.
12. Reopen the drawer and repeat key variants below (negative and edge scenarios).

#### Expected result
- Create Proposal cannot proceed when Time Zone is missing.
- User gets clear, field-level required validation for Time Zone.
- After valid Time Zone selection, validation clears and submit can proceed.

#### Negative and edge scenarios to execute under same ID
- **N1 - Time Zone + Proposal Name both missing**
  - Keep both fields empty and submit.
  - Expected: form blocked; both validation states discoverable or focus moves consistently to first invalid field.

- **N2 - Toggle clear after valid selection**
  - Select Time Zone, then clear/reset it, then submit.
  - Expected: Time Zone treated empty again; required validation reappears; submit blocked.

- **N3 - Contract Dates checkbox interaction**
  - Check `Contract Dates to be decided` (date fields hidden), keep Time Zone missing, submit.
  - Expected: submit still blocked due to missing Time Zone.

- **N4 - Multiple rapid submits**
  - Click `Create Proposal` repeatedly with missing Time Zone.
  - Expected: no duplicate stacked errors; no UI freeze; stable single validation state.

- **N5 - Cancel and reopen after error**
  - Trigger Time Zone error, click `Cancel`, reopen drawer.
  - Expected: stale error not persisted incorrectly; validation appears only after new invalid submit.

- **N6 - Keyboard-only interaction**
  - Use only keyboard (Tab/Shift+Tab/Enter) to trigger validation with missing Time Zone.
  - Expected: blocking behavior matches mouse flow; focus/error remains accessible.

- **N7 - Time Zone list usability and persistence**
  - Open Time Zone list, choose non-default option, close/reopen selector before submit.
  - Expected: chosen value remains until user changes/clears it.

- **N8 - False positive protection**
  - Fill all required fields including valid Time Zone and submit.
  - Expected: no false `Time Zone required` error.

## Performance Expectations

### Execution Timeline

| Scenario | Setup Time | Per-Test Average | Total Suite Time |
|----------|-----------|------------------|------------------|
| **First Run** (create company/property) | ~60–90s | ~15–20s | ~8–10 min |
| **Subsequent Runs** (reuse PAT resources) | ~5–10s | ~15–20s | ~7–8 min |
| **Repeated Runs** (fully cached) | <1s | ~15–20s | ~6–7 min |

### Single-Session Benefits

- ✅ **60% faster setup**: Reuse PAT-prefixed company/property instead of creating new ones
- ✅ **~90s saved per run**: No redundant auth, no per-test session overhead
- ✅ **Consistent state**: Shared browser session eliminates flaky navigation issues
- ✅ **Scalable**: 62 tests run in one continuous session instead of breaking into multiple

### Tips for Maximum Performance

1. **Run Full Suite**: Use `npm run test:uat:contract` to leverage single-session efficiency
2. **Keep PAT Resources**: The test suite searches for and reuses existing PAT-prefixed resources—no need to clean them up between runs
3. **Skip Smoke Tests for Iterations**: When debugging E2E or deletion tests, use `--grep "TC-CONTRACT-E2E|TC-CONTRACT-DELETE"` to skip the 22 smoke tests
4. **Parallel Execution**: The suite is serial by design (single worker); do not enable parallel workers (would break shared state)

### When to Expect Longer Times

- **Network latency**: Slow internet → longer `networkidle` waits
- **Auth0 delays**: Expired credentials or service outage → login retry overhead (2x)
- **First run ever**: New company/property creation + auth → ~10 minutes total
- **Headless false**: Visualizing browser adds ~1–2 minutes per run

## Architecture Notes

### Why Single Session?

- **Before**: Each `test.describe` had its own session → 3 logins for 62 tests → ~3 minutes overhead
- **After**: One login per entire suite → ~15 seconds overhead
- **Result**: ~90 seconds saved per full run (60% faster)

### How It Works

1. **`beforeAll()` (runs once)**:
   - Authenticate user
   - Create browser context
   - Initialize page objects
   - Create/resolve deal
   - Mark session ready ✓

2. **`beforeEach()` (runs before each test)**:
   - Await shared `setupPromise`
   - Check current URL
   - Navigate to Deals only if needed
   - Proceed with test ✓

3. **`afterAll()` (runs once, after all tests)**:
   - Close browser context
   - Cleanup auth file ✓

### PAT Resource Reuse Flow

```
Test Run 1:
  └─ Search for PAT company → NOT FOUND
  └─ Create PAT_12345
  └─ Search for PAT property → NOT FOUND
  └─ Create PAT_12346
  └─ Create deal, run 62 tests ✓

Test Run 2 (immediately after):
  └─ Search for PAT company → FOUND: PAT_12345 ✓
  └─ Search for PAT property → FOUND: PAT_12346 ✓
  └─ Create NEW deal (always fresh), run 62 tests ✓

Test Run N:
  └─ Reuse PAT_12345 + PAT_12346 ✓
  └─ Reuse entire session ✓
  └─ ~7–8 minutes total ✓
```
